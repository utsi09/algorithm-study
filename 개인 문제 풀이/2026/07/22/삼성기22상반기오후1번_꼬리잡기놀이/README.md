## 문제 풀이

처음에는 각 팀의 꼬리 좌표를 저장해두고, 꼬리부터 BFS를 돌면서 사람들의 번호를 한 칸씩 뒤로 미는 식으로 구현했다.

꼬리였던 `3`을 시작으로 다음 사람에게 값을 넘겨주고, 머리 앞에 있는 이동선 `4`를 찾으면 그 자리에 새로운 머리 `1`을 넣으려고 했다.

```cpp
int tmp = adj[ny][nx];
adj[ny][nx] = value;
value = tmp;
```

이런 식으로 현재 들고 있는 값을 다음 칸에 넣고, 그 칸의 기존 값을 다시 들고 이동했다.

처음에는 이동선이 비어 있는 일반적인 경우만 생각했다.

```text
3 → 2 → 2 → 1 → 4
```

꼬리부터 값을 밀면

```text
4 → 3 → 2 → 2 → 1
```

이 되는 식이다.

그런데 사람이 이동선을 전부 차지하고 있어서 머리와 꼬리가 붙어 있는 경우가 있었다.

```text
1 - 2 - 2 - 3
|           |
-------------
```

이 경우에는 머리 앞에 `4`가 없고, 기존 꼬리 자리로 머리가 이동해야 했다.

그래서 꼬리에서 탐색을 시작했을 때 바로 옆에 머리 `1`이 있으면 사람이 이동선을 꽉 채운 경우라고 판단했다.

처음에는 아래처럼 작성했다.

```cpp
if(value == 3 && adj[ny][nx] == 1){
    is_ring = true;
}
```

그런데 어떤 팀은 제대로 판정되고 어떤 팀은 판정되지 않았다.

원인은 `value`가 현재 칸의 실제 번호가 아니라, 다음 칸에 넘겨줄 값을 저장하는 변수였기 때문이다.

꼬리에서 인접한 칸을 확인할 때 몸통 `2`를 먼저 만나면

```cpp
value = 2;
```

로 바뀌었다.

그 뒤에 머리 `1`을 확인하더라도 이미 `value == 3`이 아니기 때문에 사람이 이동선을 꽉 채운 경우라는 판정이 되지 않았다.

즉, 상하좌우를 어떤 순서로 확인하는지에 따라 결과가 달라지고 있었다.

이 부분은 `value`를 보는 것이 아니라, 현재 탐색 중인 좌표가 처음 시작한 꼬리 좌표인지 확인해야 했다.

```cpp
bool is_start_tail =
    (ii == tail_r && jj == tail_c);

if(is_start_tail && adj[ny][nx] == 1){
    is_full = true;
}
```

이번에 가장 오래 걸린 이유는 하나의 변수를 두 가지 의미로 사용했기 때문인 것 같다.

```text
value
1. 현재 위치의 사람 번호
2. 다음 칸에 넘겨줄 사람 번호
```

처음에는 둘이 같은 값처럼 보여서 문제없다고 생각했는데, 탐색 도중 값이 변경되면서 전혀 다른 의미가 됐다.

다음부터 시뮬레이션에서 값이 이상하게 바뀌면 바로 아래를 확인해야 할 것 같다.

```text
이 변수가 현재 상태를 나타내는지
다음 상태를 만들기 위한 임시 값인지
```

둘을 동시에 나타내게 만들면 디버깅할 때 헷갈리기 쉽다.

---

공을 맞은 사람이 머리부터 몇 번째 사람인지 구할 때도 처음에는 BFS 방문 거리를 사용했다.

```cpp
visited[ny][nx] = visited[ii][jj] + 1;
```

일반적인 팀에서는 머리부터 사람을 따라가는 경로가 하나이기 때문에 이 방법이 맞아 보였다.

하지만 사람이 이동선을 전부 채우면 머리와 꼬리가 서로 붙어 있다.

이때 공이 꼬리를 맞으면 BFS는 머리와 꼬리가 바로 연결돼 있다고 판단해서 거리를 짧게 계산한다.

실제로 필요한 것은 격자에서의 최단거리가 아니라 팀 안에서의 순서다.

```text
머리 1
→ 몸통 2
→ 몸통 2
→ 꼬리 3
```

따라서 머리부터 시작해서 이전에 방문한 사람을 제외하고 `2`, 마지막에는 `3`을 따라가야 한다.

이 문제는 격자 문제처럼 보이지만 사람들만 놓고 보면 연결 리스트와 비슷했다.

```text
머리 → 두 번째 사람 → 세 번째 사람 → ... → 꼬리
```

무조건 BFS를 써야 하는 것이 아니라, 한 줄로 연결된 구조라면 다음 사람을 하나씩 따라가는 방식이 더 정확했다.

---

공이 날아오는 방향도 네 방향을 따로 구현했다.

```text
0 : 왼쪽에서 오른쪽
1 : 아래에서 위
2 : 오른쪽에서 왼쪽
3 : 위에서 아래
```

라운드를 `n`으로 나눈 구간에 따라 방향을 정했다.

```cpp
int turn = (t / n) % 4;
int line = t % n;
```

여기서 마지막 방향을 아래에서 위로 작성하는 실수도 있었다.

시뮬레이션 문제에서는 방향을 머릿속으로만 생각하면 반대로 구현하기 쉬운 것 같다.

다음부터는 코드부터 작성하지 말고 먼저 아래처럼 적고 시작해야겠다.

```text
방향 0
행: 0 → n-1
공: 왼쪽 → 오른쪽

방향 1
열: 0 → n-1
공: 아래 → 위

방향 2
행: n-1 → 0
공: 오른쪽 → 왼쪽

방향 3
열: n-1 → 0
공: 위 → 아래
```

## 이번 문제에서 오래 걸린 부분

처음 설계 자체가 완전히 틀린 것은 아니었다.

꼬리 좌표를 팀별로 저장하고, 팀을 이동시킨 뒤 공을 던지고, 맞은 팀의 머리와 꼬리를 바꾸는 전체 흐름은 맞았다.

오래 걸린 이유는 예외 상황을 처리하면서 기존 변수의 의미가 계속 섞였기 때문이다.

특히 아래 세 부분에서 많이 꼬였다.

```text
1. 사람이 이동선을 전부 채운 경우
2. BFS 최단거리와 팀 내 순서를 같은 것으로 생각한 것
3. 방향에 따라 탐색 순서가 달라지는 것을 놓친 것
```

한 팀에서는 되고 다른 팀에서는 안 되면 처음에는 입력 모양이나 링 형태가 다른 줄 알았다.

그런데 실제 원인은 상하좌우 탐색 순서였다.

이런 경우에는 전체 맵을 계속 출력하는 것보다 변수 하나가 언제 바뀌는지 보는 것이 더 중요했다.

```cpp
cout << ii << ", " << jj
     << " value: " << value << '\n';
```

특히 조건문에 사용되는 값이 루프 중간에 바뀌고 있는지 먼저 확인해야 한다.

## 다음에 비슷한 문제를 만나면

시뮬레이션 문제에서는 먼저 각 숫자의 의미를 확실히 분리해야겠다.

```text
0 : 아무것도 없음
1 : 머리
2 : 몸통
3 : 꼬리
4 : 이동선
```

그다음 한 라운드에서 변하는 순서를 함수로 나눈다.

```text
1. 모든 팀 이동
2. 공 발사
3. 맞은 사람의 순서 계산
4. 점수 추가
5. 머리와 꼬리 교환
```

그리고 변수 하나에 여러 의미를 넣지 않는다.

```text
현재 칸의 값
다음 칸에 넣을 값
현재 좌표
처음 시작 좌표
```

이런 값들은 따로 관리해야 한다.

또 격자에서 탐색한다고 무조건 BFS 거리로 답을 구하면 안 된다.

문제에서 요구하는 것이

```text
최단거리인지
연결된 순서인지
이동 횟수인지
```

먼저 구분해야 한다.

이번 문제의 사람들은 격자에 있지만 실제 구조는 한 줄로 연결된 팀이었다.

그래서 팀 내 순번은 BFS 최단거리가 아니라 머리부터 순서대로 따라가야 했다.

마지막으로 특정 모양에서만 틀리면 그 모양만 계속 쳐다보지 말고, 상하좌우 탐색 순서가 달라졌을 때도 같은 결과가 나오는지 확인해야겠다.

이번에는 구현량이 많아서 오래 걸린 것도 있지만, 거의 다 만든 뒤에 상태 변수 하나의 의미가 섞여서 시간을 많이 썼다.

다음에는 코드가 길어질수록 각 변수가 정확히 무엇을 의미하는지 주석으로 먼저 적고 구현해야겠다.
